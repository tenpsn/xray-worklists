# ระบบรายงานผล X-ray + เชื่อมต่อเครื่อง Modality (DICOM Worklist)

## ระบบนี้ทำอะไร

ระบบนี้ทำหน้าที่เป็น **ตัวกลาง** ระหว่างฐานข้อมูล HIS (ระบบโรงพยาบาล เช่น HOSXP) กับเครื่อง X-ray/CT/MRI (เรียกรวมว่า "เครื่อง Modality") โดยทำ 2 อย่างหลักๆ:

1. **แสดงรายงานผล X-ray แบบ realtime บนหน้าเว็บ** — ดึงรายชื่อผู้ป่วยที่ถูกส่งตรวจ X-ray จากฐานข้อมูล HIS มาแสดงเป็นตาราง พร้อมกรอง/ค้นหาได้ และ auto-refresh ทุก 10 วินาที
2. **สร้างไฟล์ Worklist ส่งให้เครื่อง Modality อัตโนมัติ** — ทุกครั้งที่มีเคสใหม่หรือข้อมูลเปลี่ยน ระบบจะแปลงข้อมูลนั้นเป็นไฟล์ DICOM Modality Worklist (`.wl`) เพื่อให้เครื่อง X-ray ดึงชื่อ-ข้อมูลผู้ป่วยไปกรอกอัตโนมัติ (ไม่ต้องพิมพ์เองที่หน้าเครื่อง ลดความผิดพลาด)

โปรเจกต์แยกเป็น 2 ส่วน รันคนละ process:

```
xray-worklists/
├── backend/    -> Express API (port 4000) — คุยกับ HIS DB + จัดการ DICOM
└── frontend/   -> Next.js (port 3000) — หน้าเว็บแสดงตาราง + หน้าตั้งค่า
```

## ระบบสร้าง / แก้ไข / ลบ อะไรบ้าง (ไฟล์และข้อมูล)

ระบบนี้ไม่ได้แก้ไขข้อมูลใน HIS DB โดยตรง (การยืนยันผล/อ่านฟิล์ม ยังต้องทำที่ HIS ตามปกติ) แต่จะ **สร้าง อัปเดต และลบไฟล์บน server ของตัวเองอัตโนมัติ** ดังนี้:

### 1) โฟลเดอร์ `backend/worklists/`
| ไฟล์ | เกิดขึ้นเมื่อไร | ถูกลบเมื่อไร |
|---|---|---|
| `<XN>.dump` | สร้างชั่วคราวทุกครั้งที่มีเคสใหม่/ข้อมูลเปลี่ยน เพื่อเตรียมแปลงเป็น `.wl` | ลบทันทีอัตโนมัติหลังแปลงเป็น `.wl` สำเร็จ (ถ้าไฟล์ถูกล็อกอยู่ ระบบจะลองลบซ้ำสูงสุด 3 ครั้ง) |
| `<XN>.wl` | สร้าง/อัปเดตด้วยคำสั่ง `dump2dcm` (จาก DCMTK) ทุกครั้งที่ข้อมูลผู้ป่วยรายนั้นเปลี่ยนไปจากครั้งก่อน (เทียบด้วย hash — ถ้าไม่เปลี่ยนจะข้าม ไม่สร้างซ้ำ) | ลบอัตโนมัติเมื่อ **(ก)** สถานะในตารางกลายเป็น "ยืนยันผลแล้ว" ทั้งคู่ (`confirm = Y` และ `confirm_read_film = Y`) หรือ **(ข)** เครื่อง Modality ส่งสถานะ MPPS กลับมาว่า `COMPLETED` หรือ `DISCONTINUED` (ตรวจเสร็จ/ยกเลิก) |
| `.worklist-state.json` | เก็บ hash ข้อมูลล่าสุด + StudyInstanceUID ของแต่ละ XN เพื่อไม่ให้สุ่ม UID ใหม่ทุกครั้งและไม่สร้างไฟล์ซ้ำโดยไม่จำเป็น | ลบ entry ของ XN นั้นออกจากไฟล์นี้เมื่อ `.wl` ของ XN นั้นถูกลบ (ไฟล์ state เองไม่ถูกลบทั้งไฟล์) |

### 2) โฟลเดอร์ `backend/logs/`
| ไฟล์ | เกิดขึ้นเมื่อไร | ถูกลบเมื่อไร |
|---|---|---|
| `dicom-network-YYYY-MM-DD.log` | สร้างวันละไฟล์ เก็บ log การเชื่อมต่อ DICOM (MPPS) จากเครื่อง Modality | ลบอัตโนมัติเมื่ออายุไฟล์เกิน **7 วัน** (เช็คทุกครั้งที่ขึ้นวันใหม่) |

### 3) `backend/mpps-state.json`
เก็บ mapping ระหว่าง SOP Instance UID กับ Accession Number (XN) ที่เครื่อง Modality ส่ง N-CREATE (เริ่มตรวจ) เข้ามา เพื่อให้จับคู่ได้ถูกต้องตอนได้รับ N-SET (ตรวจเสร็จ/ยกเลิก) ทีหลัง — ระบบจะลบ entry นั้นออกทันทีที่ได้รับสถานะ `COMPLETED`/`DISCONTINUED`

### 4) `backend/settings.json`
เก็บค่าที่ตั้งจากหน้า "ตั้งค่าระบบ" (การเชื่อมต่อ HIS DB + MWL) แทนที่ค่าจาก `.env` เดิม — รหัสผ่านจะไม่ถูกส่งกลับไปแสดงที่หน้าเว็บ (Backend จะปิดบังด้วย `••••••••` เสมอ) และจะไม่ถูกเขียนทับด้วยค่าว่างโดยไม่ตั้งใจตอนกดบันทึก

**สรุป:** ระบบไม่ลบ/แก้ไขอะไรใน HIS DB ทั้งสิ้น สิ่งที่ระบบสร้าง-ลบเองอัตโนมัติมีแค่ไฟล์ worklist (`.dump`/`.wl`), ไฟล์ log, และไฟล์ state/settings ภายในเครื่อง backend เท่านั้น

## ส่วนประกอบของระบบ

### Backend

| ไฟล์ | หน้าที่ |
|---|---|
| `server.js` | จุดเริ่มต้นของ backend เปิด API ให้ frontend เรียก, ดึงข้อมูลจาก HIS DB, สั่งสร้างไฟล์ worklist |
| `db.js` | ตัวเชื่อมต่อฐานข้อมูล รองรับทั้ง **Postgres และ MySQL** สลับกันได้ตามค่าที่ตั้งไว้ในหน้า Settings (จัดการเรื่อง encoding ภาษาไทย TIS-620/Windows-874/UTF-8 ให้ด้วย) |
| `dicomService.js` | แปลงข้อมูลผู้ป่วย 1 รายการ ให้เป็นไฟล์ DICOM Worklist (`.wl`) ที่เครื่อง Modality อ่านได้ — รวมถึงแปลงชื่อไทยเป็นคาราโอเกะ ถ้าเลือกภาษาอังกฤษ ต้องใช้ `dump2dcm.exe` (จาก DCMTK) ในการแปลง `.dump` เป็น `.wl` |
| `mppsService.js` | เปิด server แยกต่างหาก (พอร์ต 7001) รอรับสถานะจากเครื่อง Modality ว่า "เริ่มตรวจ" (N-CREATE) หรือ "ตรวจเสร็จ/ยกเลิก" (N-SET) — พอตรวจเสร็จจะลบไฟล์ worklist ทิ้ง ไม่ให้เครื่องดึงเคสเดิมไปทำซ้ำ |
| `settingsService.js` | โหลด/บันทึกการตั้งค่าการเชื่อมต่อ HIS DB และ MWL (Modality Worklist) ลงไฟล์ `settings.json` |

### Frontend

| ไฟล์ | หน้าที่ |
|---|---|
| `app/page.js` | หน้าหลัก แสดงตารางรายงานผล X-ray, มีตัวกรอง (ย้อนหลังกี่วัน, include/exclude, เฉพาะที่ยังไม่ยืนยัน), auto-refresh ทุก 10 วิ, สลับภาษาไทย/อังกฤษได้ |
| `app/settings/page.js` | หน้าตั้งค่าการเชื่อมต่อ HIS DB (host, port, username, password, encoding) และตั้งค่า MWL (AE Title, port) |
| `app/lib/nameDisplay.js` | ฟังก์ชันช่วยจัดรูปแบบชื่อ-นามสกุล-คำนำหน้าที่แสดงในตาราง ตัดความยาว, แปลงเป็นคาราโอเกะตอนเลือกภาษาอังกฤษ |

## ฟีเจอร์หลัก

- แสดงรายชื่อผู้ป่วยที่ถูกสั่งตรวจ X-ray แบบ auto-refresh ทุก 10 วินาที
- กรองข้อมูลได้: ย้อนหลังกี่วัน, ค้นหาชื่อรายการตรวจ (include/exclude), แสดงเฉพาะที่ยังไม่ยืนยันผล
- สลับแสดงผลภาษาไทย ↔ อังกฤษ (แปลงชื่อเป็นคาราโอเกะอัตโนมัติ) โดยไม่กระทบข้อมูลจริงใน DB
- สร้างไฟล์ DICOM Modality Worklist ให้เครื่อง X-ray อัตโนมัติ ไม่ต้องพิมพ์ข้อมูลผู้ป่วยเองที่เครื่อง พร้อมข้ามการสร้างซ้ำถ้าข้อมูลไม่เปลี่ยน
- รับสถานะตรวจเสร็จจากเครื่อง Modality (MPPS) แล้วเคลียร์ไฟล์ worklist ให้เองอัตโนมัติ พร้อมทั้งลบไฟล์เมื่อยืนยันผลครบทั้งสองสถานะจาก HIS
- รองรับฐานข้อมูล HIS ทั้ง Postgres และ MySQL สลับกันได้จากหน้า Settings
- ตั้งค่าการเชื่อมต่อทั้งหมด (DB + DICOM) ผ่านหน้าเว็บ พร้อมทดสอบเชื่อมต่อจริงทันทีที่บันทึก
- เก็บ log การเชื่อมต่อ DICOM รายวัน และลบไฟล์ log เก่าเกิน 7 วันให้อัตโนมัติ

## สิ่งที่ต้องเตรียมก่อนติดตั้ง (Prerequisites)

ก่อนติดตั้งครั้งแรก ให้เตรียมสิ่งเหล่านี้ให้พร้อม:

1. **Node.js** เวอร์ชัน 18 ขึ้นไป และ `npm` (ติดตั้งมาพร้อมกัน)
2. **เข้าถึงฐานข้อมูล HIS ได้** (Postgres หรือ MySQL) — ต้องรู้ IP, Port, ชื่อฐานข้อมูล, username, password และตารางต้องมีคอลัมน์ตามที่ query ใน `server.js` ต้องการ (เช่น ตาราง `xray_report`, `patient`, `xray_items`, `doctor`, `xray_head`)
3. **DCMTK** — โปรแกรมสำหรับแปลงไฟล์ `.dump` เป็น `.wl` ต้องมีไฟล์ `dump2dcm.exe` วางไว้ที่ `backend/dcmtk/bin/dump2dcm.exe` (ปัจจุบัน path เขียนแบบ Windows เท่านั้น จึงต้องรันบนเครื่อง Windows หรือแก้ path ในโค้ดถ้าจะรันบน Linux/Mac)
4. **PM2** (สำหรับ production) — ใช้จัดการให้ backend และ frontend รันต่อเนื่องอยู่เบื้องหลัง

## วิธีติดตั้งระบบครั้งแรก (First-time setup)

### ขั้นที่ 1 — วางโค้ดลงเครื่อง

นำโฟลเดอร์โปรเจกต์ทั้งหมดไปวางไว้ตำแหน่งที่ต้องการ เช่น `D:\xray-worklists` (ตำแหน่งนี้ต้องตรงกับที่ระบุใน `start-system.bat`)

```
xray-worklists/
├── backend/
├── frontend/
├── ecosystem.config.js
└── start-system.bat
```

### ขั้นที่ 2 — เตรียม DCMTK

สร้างโฟลเดอร์ `backend/dcmtk/bin/` แล้วนำไฟล์ `dump2dcm.exe` (และ dll ที่จำเป็น) จากชุดติดตั้ง DCMTK ไปวางไว้ในนั้น

### ขั้นที่ 3 — ติดตั้งและตั้งค่า Backend

```bash
cd backend
npm install
copy .env.example .env
```

เปิดไฟล์ `.env` แล้วกรอกค่าเบื้องต้นอย่างน้อย:
- `PORT` (ค่าเริ่มต้นแนะนำ 4000)
- `CORS_ORIGIN` (ต้องตรงกับ URL ของ frontend เช่น `http://localhost:3000`)
- ค่าเชื่อมต่อฐานข้อมูล HIS เบื้องต้น (`DB_TYPE`, `PGHOST`, `PGPORT`, `PGDATABASE`, `PGUSER`, `PGPASSWORD`, `DB_ENCODING`) — ค่านี้เป็นแค่ **ค่าเริ่มต้น** เผื่อยังไม่เคยเข้าไปตั้งผ่านหน้าเว็บ ทีหลังสามารถเข้าไปแก้ที่หน้า "ตั้งค่าระบบ" ได้เลยโดยไม่ต้องแก้ `.env` อีก

ทดสอบรัน backend:

```bash
npm run dev
```

เช็คว่าตื่นอยู่: เปิดเบราว์เซอร์ไปที่ `http://localhost:4000/health` ควรเห็น `{"ok":true}`

### ขั้นที่ 4 — ติดตั้งและตั้งค่า Frontend

เปิด terminal อีกอันหนึ่ง:

```bash
cd frontend
npm install
copy .env.local.example .env.local
```

เปิดไฟล์ `.env.local` แล้วตั้งค่า `NEXT_PUBLIC_API_URL` ให้ชี้ไปที่ backend เช่น `http://localhost:4000`

รัน frontend:

```bash
npm run dev
```

เปิด `http://localhost:3000` — ถ้าขึ้นตารางรายงานผล (แม้จะยังว่างหรือ error เพราะยังไม่ได้ตั้งค่า DB) แปลว่าเชื่อมต่อกันได้แล้ว

### ขั้นที่ 5 — ตั้งค่าการเชื่อมต่อ HIS DB และ MWL ผ่านหน้าเว็บ

เข้าเมนู **"ตั้งค่าระบบ"** ที่มุมขวาบนของหน้าเว็บ แล้วกรอก:
- **HIS (HOSXP) — Database:** เลือกประเภทฐานข้อมูล (MySQL/Postgres), IP, Port, ชื่อฐานข้อมูล, Username, Password, Encoding ให้ตรงกับฐานข้อมูลจริง
- **MWL — DICOM Modality Worklist:** AE Title, Port สำหรับรับ C-FIND จากเครื่อง Modality, Port สำหรับรับ MPPS, และ **โฟลเดอร์เก็บไฟล์ Worklist (.wl)**

กด **"บันทึกการตั้งค่า"** — ระบบจะทดสอบเชื่อมต่อฐานข้อมูลให้ทันทีและแจ้งผลว่าสำเร็จหรือไม่

#### เลือกโฟลเดอร์เก็บไฟล์ Worklist ได้

ปกติไฟล์ `.wl` จะถูกเก็บไว้ในโฟลเดอร์ `backend/worklists` ของระบบเอง แต่ถ้าต้องการให้โฟลเดอร์นี้อยู่ **นอก backend** เช่น อยู่บนไดรฟ์ที่แชร์ร่วมกับเครื่อง Orthanc (Worklist Server) โดยตรง ในหน้า Settings มีปุ่ม **"เลือกโฟลเดอร์..."** ให้กดเปิดหน้าต่างเดินดูโฟลเดอร์จริงบนเครื่อง backend ได้เลย คลิกเข้าไปในโฟลเดอร์ที่ต้องการ หรือกด "+ สร้างโฟลเดอร์" เพื่อสร้างโฟลเดอร์ใหม่ในตำแหน่งนั้น แล้วกด "ใช้โฟลเดอร์นี้" เพื่อยืนยัน

- ถ้าไม่เลือก (ค่าว่าง) ระบบจะใช้ค่า default คือ `backend/worklists` อัตโนมัติ — มีปุ่ม "ใช้ค่าเริ่มต้น" ให้กดล้างค่ากลับได้
- เปลี่ยนได้ตลอดเวลาโดยไม่ต้อง restart backend เอง — บันทึกจากหน้าเว็บแล้วมีผลทันที
- **สำคัญ:** เมื่อเปลี่ยนโฟลเดอร์นี้แล้ว **ต้องไปแก้ค่า `"Worklists"."Database"` ในไฟล์ตั้งค่าของ Orthanc (`orthanc.json`) ให้ชี้มาที่ path เดียวกันด้วยเสมอ** ไม่งั้นเครื่อง Modality จะหาไฟล์ worklist ไม่เจอ (ดูหัวข้อ "ตั้งค่า Orthanc" ด้านล่าง)

### ขั้นที่ 6 — ตั้งค่าเครื่อง Modality (X-ray/CT/MRI)

ที่ฝั่งเครื่อง Modality ให้ตั้งค่า Worklist Server ให้ชี้มาที่ IP ของเครื่อง backend นี้ พร้อม Port และ AE Title ให้ตรงกับที่ตั้งไว้ในขั้นที่ 5 (MWL Port สำหรับดึงรายชื่อ, MPPS Port สำหรับส่งสถานะกลับ)

### ขั้นที่ 7 — ตั้งค่า Orthanc (ตัว DICOM Worklist Server ที่แท้จริง)

Orthanc คือโปรแกรมที่ทำหน้าที่ตอบ C-FIND จากเครื่อง Modality จริงๆ โดยอ่านไฟล์ `.wl` ที่ backend สร้างไว้ ต้องแก้ไฟล์ config `orthanc.json` อย่างน้อยตามนี้ตอนติดตั้งครั้งแรก:

| ค่าที่ต้องแก้ | ต้องเป็น |
|---|---|
| `StorageDirectory`, `IndexDirectory` | โฟลเดอร์ที่มีอยู่จริงบนเครื่องนั้น (สร้างไว้ล่วงหน้า) |
| `Plugins` | path ไปยังโฟลเดอร์ Plugins ตามตำแหน่งที่ติดตั้ง Orthanc จริง |
| `Worklists.Database` | **ต้องตรงกับโฟลเดอร์ worklists ที่ backend ใช้งานอยู่จริง** (ค่า default คือ `backend/worklists` หรือ path ที่ตั้งไว้เองในขั้นที่ 5 ด้านบน) |
| `DicomAet` | ต้องตรงกับค่า **AET** ที่ตั้งไว้ในหน้า Settings ของเว็บแอป (ค่า default คือ `ORTHANC` ทั้งคู่) |
| `DicomPort` | **ต้องตรงกับค่า Port** ที่ตั้งไว้ในหน้า Settings ของเว็บแอป — ค่า default ของ Orthanc คือ `4242` แต่ค่า default ของเว็บแอปคือ `7000` เลือกให้ตรงกันทางใดทางหนึ่ง แล้วตั้งเครื่อง Modality ให้ใช้พอร์ตเดียวกันนี้ด้วย |

หลังแก้ไฟล์ `orthanc.json` แล้ว ต้อง restart โปรแกรม Orthanc ให้ค่าที่แก้มีผล (Orthanc ไม่ได้ auto-reload config เหมือนหน้าเว็บของระบบนี้)

## การใช้งานจริง (Production)

แนะนำให้รันทั้ง backend และ frontend ต่อเนื่องด้วย PM2 แทนการรัน `npm run dev` ค้างไว้:

```bash
# build frontend ก่อน (ครั้งแรกและทุกครั้งที่แก้โค้ด)
cd frontend
npm run build
cd ..

# ติดตั้ง pm2 (ถ้ายังไม่มี)
npm install pm2

# สั่งรันทั้งสองระบบพร้อมกันด้วย ecosystem.config.js
.\node_modules\.bin\pm2 start ecosystem.config.js
```

หรือใช้ไฟล์ `start-system.bat` ที่เตรียมไว้ให้ (ตั้งค่าให้รันอัตโนมัติตอนเปิดเครื่อง Windows ได้ด้วย Task Scheduler) — ไฟล์นี้จะ `cd` ไปที่ `D:\xray-worklists` แล้วสั่ง `pm2 start ecosystem.config.js` ให้เอง ถ้าวางโปรเจกต์ไว้ตำแหน่งอื่น ต้องแก้ path ในไฟล์นี้ให้ตรงด้วย

ตรวจสอบสถานะ / log ของ PM2:

```bash
pm2 status
pm2 logs xray-backend
pm2 logs xray-frontend
```

### ความทนทานต่อ error (Reliability)

ระบบตั้งค่าไว้ ดังนี้:

- **`max_memory_restart`** ใน `ecosystem.config.js` — ถ้า backend กินแรมเกิน `300M` หรือ frontend เกิน `400M` (เช่นเกิด memory leak สะสม) PM2 จะสั่ง restart process นั้นให้อัตโนมัติก่อนที่จะกระทบเครื่อง server ทั้งเครื่อง ปรับค่าได้ตามการใช้งานจริงหลังขึ้นระบบ
- **Uncaught exception / unhandled rejection** — ถ้าเกิด error ที่ไม่ได้ดักไว้ทั่วทั้งโปรแกรม ระบบจะ log แล้ว **ปิดโปรแกรมทันที** (`process.exit(1)`) แทนที่จะพยายามทำงานต่อในสภาพที่ state อาจเพี้ยนไปแล้ว แล้วปล่อยให้ PM2 (`autorestart: true`) เริ่ม process ใหม่แบบสะอาด
- **Port ชนกัน** — ถ้า backend หลัก (port ตาม `.env`) เปิดไม่สำเร็จเพราะมีโปรแกรมอื่นใช้ port นั้นอยู่แล้ว (เช่นรันซ้อนกัน 2 รอบ) ระบบจะปิดโปรแกรมทันทีพร้อม log ชัดเจน แทนที่จะแกล้งทำเป็นทำงานได้ปกติทั้งที่เข้าใช้งานจริงไม่ได้เลย
  - ส่วน MPPS server (port แยกต่างหาก) ถ้า **ตอนสตาร์ทเครื่อง** เปิดไม่สำเร็จ จะถือว่าร้ายแรงและปิดโปรแกรมเช่นกัน แต่ถ้า **เปลี่ยน MPPS port จากหน้าเว็บ Settings** แล้วเปิดไม่สำเร็จ (เช่น port ชนกับโปรแกรมอื่น) จะแค่แจ้งเตือนในข้อความหลังบันทึก ไม่ปิดทั้งระบบ เพราะ API หลักส่วนอื่นยังใช้งานได้ปกติ
- **`GET /health`** ตอนนี้ทดสอบเชื่อมต่อฐานข้อมูลจริงด้วย (`SELECT 1`) ไม่ใช่แค่ตอบว่า process ยังไม่ตาย — ถ้าต่อ DB ไม่ได้จะตอบ HTTP `503` กลับไป เหมาะสำหรับให้เครื่องมือ monitor ภายนอกเรียกเช็คเป็นระยะ

- ระบบไม่แก้ไข/ลบข้อมูลใดๆ ในฐานข้อมูล HIS โดยตรง การยืนยันผล/อ่านฟิล์ม ยังคงต้องทำผ่านระบบ HIS ตามปกติ

- `dump2dcm.exe` ต้องมีอยู่จริงตาม path ที่กำหนด มิฉะนั้นระบบจะสร้างได้แค่ไฟล์ `.dump` (เครื่อง Modality จะยังไม่เห็นรายการ เพราะต้องเป็นไฟล์ `.wl` เท่านั้น)
